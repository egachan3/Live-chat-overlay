# 要件・仕様

## 対応範囲（フェーズ分け）

1. **フェーズ1**：YouTube Live のみ実装・動作確認
2. **フェーズ2**：Twitch 対応を追加
3. **公開**：GitHub（OSS）＋ Chrome Web Store
4. **公開後**：Kick 対応を追加

## 確定した仕様

| 項目 | 決定内容 |
|---|---|
| 対応サイト（最終） | YouTube Live + Twitch（Kickは公開後に追加） |
| YouTube対応範囲 | ライブ配信中 ＋ アーカイブのチャットリプレイ（同じDOM構造 `ytd-live-chat-frame` を使うため実質同時対応） |
| 通常動画コメント欄 | 対象外（別DOM構造・非リアルタイムのため） |
| 表示スタイル | 積み上げ型／横スクロール型（ニコニコ動画の弾幕表示に近い、右から左へ流れる表示）の2種類。ポップアップから切り替え可能 |
| 積み上げ型の表示位置 | 動画の左側／右側をポップアップから選択可能（デフォルトは左側） |
| 横スクロール型の表示範囲 | 動画の横幅いっぱいを使用。弾幕方式のレーン管理（空いている行から表示、一定時間で画面を横切る） |
| 投稿者名の表示 | 表示しない（コメント本文のみ）。データ取得自体は行っている |
| 設定保存 | `chrome.storage.local`（端末ローカル、ブラウザ・PC再起動後も保持、アンインストールまで維持） |
| ON/OFF制御 | ポップアップ内トグル。グローバルな1スイッチ（タブごとの個別制御はしない） |
| ポップアップ構成 | ON/OFFトグル・文字サイズ・表示スタイル・積み上げ型の表示位置を1画面にまとめる |
| 表示条件 | トグルONなら常時表示（全画面かどうか・ネイティブチャット表示有無を問わない） |
| 全画面対応 | `fullscreenchange` イベントを監視し、オーバーレイ要素を全画面要素の子に再配置（Fullscreen APIの制約対応） |
| マウス操作 | `pointer-events: none` で完全クリック透過。動画操作を一切妨げない |
| 積み上げ型の幅 | 動画幅の20〜25% |
| 積み上げ方向 | 下から積み上がり、下端固定（ネイティブチャットと同じ挙動） |
| 古いコメントの削除 | 即座に削除（フェードアウト等のアニメーションなし） |
| 最大表示件数（積み上げ型） | 固定行数ではなく「高さ基準」。パネルが埋まったら古いものを自動削除 |
| コメント装飾 | 背景ボックスなし。text-shadow による縁取りのみ |
| 文字色 | 全コメント同色 |
| 文字サイズ設定 | スライダーUI、12px〜32px、デフォルト16px |
| スクロール追従 | ページスクロールに応じてオーバーレイの座標を再計算。動画が画面外に出た場合はオーバーレイを非表示にする |
| SPA遷移対応 | `document.body` への `MutationObserver` 常時監視で対応（YouTube独自イベント `yt-navigate-finish` には依存しない）。動画・配信が切り替わった際は表示中のコメントをクリアする |
| ネイティブチャット欄との関係 | ネイティブのチャット欄を閉じても、コメントの受信・表示を継続する |
| 権限（manifest） | 必要最小限。`content_scripts` の `matches` は YouTube のみに限定し、Twitch用の `host_permissions` はフェーズ2で追加 |
| 高負荷対策 | 特別なレート制限は設けない（高さ基準の自動削除で要素数は一定範囲に収まる想定） |
| ファイル構成 | 最初から `sites/` フォルダで分離（`sites/youtube.ts` など）。Twitch追加時は `sites/twitch.ts` を追加するだけで済む設計 |

## 技術スタック

| 項目 | 使用技術・方針 |
|---|---|
| 拡張機能仕様 | Manifest V3 |
| 実装言語 | TypeScript（`strict: true`） |
| コメント検知方式 | `MutationObserver` によるDOM監視（APIキー不要） |
| ビルドツール | `tsc` のみ（バンドラー・外部依存ライブラリなし） |
| 状態管理 | `chrome.storage.local` |
| UI | 素のHTML/CSS/JS（フレームワーク不使用） |
| 成果物管理 | コンパイル後の `.js` も `.ts` と併せてリポジトリにコミット（ビルド不要でそのまま読み込める状態を維持） |

## 公開方針

- **GitHub**：オープンソースとして公開（MIT License）
- **Chrome Web Store**：デベロッパー登録・プライバシーポリシー・審査を経て公開（実装が固まった後に着手）

## 既知の制約

- 各サイトのHTML構造変更で突然動かなくなる可能性がある（特にYouTube）
- Kickは公開ドキュメントが乏しく、対応時は複数セレクタのフォールバック実装が必要になる見込み
- チャット欄がページ内に表示されている状態が前提（ポップアウトチャットは対象外）

## 実装上の注意点・ハマりどころ

開発中に判明した、YouTube連携における技術的な注意点。今後Twitch対応やメンテナンスを行う際の参考情報。

- **content scriptの「isolated world」**：Chrome拡張機能のcontent scriptは、ページ本来のJavaScriptとは隔離された実行環境（isolated world）で動作する。この隔離はフレームごとに独立しているため、同一オリジンであっても、iframe側のcontent scriptからトップフレーム側content scriptが定義したグローバル変数へ直接アクセスすることはできない（常に`undefined`になる）。フレームをまたぐ通信には`window.postMessage`が必要。
- **`content_scripts`はデフォルトで最上位フレームにしか注入されない**：`manifest.json`の`content_scripts`は、`all_frames: true`を指定しない限り、そのタブの最上位フレームにしか注入されない。iframe（YouTubeのライブチャットは`live_chat`という別ドキュメントのiframe内で描画される）にもスクリプトを注入する場合は`all_frames: true`が必須。
- **`<video>`要素はSPA遷移で使い回されることがある**：YouTubeは「おすすめ動画」のクリック等で、`<video>`要素そのものは使い回したまま、配信内容だけを差し替えることがある。動画要素の同一性だけでは「別の配信に切り替わったか」を判定できないため、チャットの取得元URL（continuationトークン）の変化を判定材料にする必要がある。
- **ネイティブのチャット欄を閉じるとYouTube側が更新を停止する**：ライブ配信・アーカイブ（チャットリプレイ）を問わず、ネイティブのチャット欄（`ytd-live-chat-frame`内のiframe）が非表示になると、YouTube側がそのiframe内での内部更新処理を停止する（DOM自体は残るが新規コメントが追加されなくなる）。これを回避するため、同じURLを独立して読み込む「隠しiframe」を画面外（`position: fixed; left: -9999px`。`display: none`や`visibility: hidden`はレンダリング停止の原因になるため使わない）に常時起動し、ネイティブの開閉状態に依存せず受信し続ける設計にした。
- **アーカイブのチャットリプレイは動画の再生位置との同期が必要**：チャットリプレイ（`live_chat_replay`）は、YouTube本体がネイティブのチャットiframeに対して`{"yt-player-video-progress": 秒数}`（約250ms間隔）・`{"yt-player-state-change": 1|2|3}`（再生中/一時停止/バッファリング中）という`postMessage`を送ることで、再生位置に同期してコメントを進めている（非公開の仕様、実機調査により判明）。隠しiframeにはこの同期が届かないため、`<video>`要素の再生状態を監視し、同じ形式のメッセージを自前で生成して隠しiframeに転送する実装が必要だった。
- **チャット欄を閉じるとネイティブiframeの中身が`about:blank#blocked`になる**：ネイティブのチャット欄を閉じると、そのiframeの`contentWindow.location.href`が`"about:blank#blocked"`（末尾にフラグメント付き）に変化する。空白ページ判定は完全一致ではなく`startsWith("about:blank")`のような前方一致で行う必要がある（完全一致だとすり抜けて、隠しiframe側にまで空白URLが伝播してしまう）。
- **z-indexの重なり順はDOM順序より数値が優先される**：兄弟要素同士でz-indexの数値が明確に異なる場合、DOM上の順序に関わらず数値が大きい方が必ず上に表示される。YouTubeのヘッダー実体（`#masthead-container`）は`z-index: 2020`。
- **カスタムトグルスイッチの`label`/`input`紐付け**：見た目のスイッチ部分を`<span>`と`<input>`の兄弟要素として実装すると、視覚的なスイッチ部分をクリックしても中の`<input>`が連動しない。スイッチ全体を`<label>`にして`<input>`をネストする必要がある。
- **`module: "None"`構成でのファイル間連携**：バンドラーを使わない方針のため、`tsconfig.json`は`module: "None"`。複数の`.ts`ファイルは`import`/`export`を使わず、グローバルスコープ（`window`にぶら下げる名前空間）経由でのみ連携できる。
